# ToonSchool Agent Rules

## 1. 절대 수정 금지 영역
- 기존 /toon 에디터는 사용자가 명시적으로 요청하지 않는 한 수정하지 않는다.
- ToonEditor.tsx, A4CoverPreview.tsx, ComicPreview.tsx, SpeechBubblePanel.tsx, SpeechBubbleOverlay.tsx, ComicCutPanel.tsx는 보호한다.
- Supabase 테이블 삭제, DROP, TRUNCATE, 대량 UPDATE는 절대 실행하지 않는다.
- Gemini API, Supabase 설정, 인증 설정은 요청 없이는 수정하지 않는다.

## 2. 학생 UI 작업 원칙
- /student 관련 파일만 수정한다.
- 학생 UI는 태블릿 중심으로 설계한다.
- 초등학생용 게임형 학습 앱 톤앤매너를 유지한다.
- 흰 배경 중심의 웹폼 UI를 만들지 않는다.
- 한 화면에서 다음에 누를 버튼이 명확해야 한다.
- 글씨는 짧고 쉽고 크게 쓴다.
- 카드, 버튼, 배경, 보상, 진행률은 통일된 톤으로 만든다.

## 3. 모듈화 원칙
- 한 파일이 600행을 넘지 않게 한다.
- 400행을 넘기면 분리 가능성을 검토한다.
- 페이지 파일에는 전체 흐름만 둔다.
- 실제 UI는 components 폴더로 분리한다.
- Supabase 조회 코드는 page 안에 직접 쓰지 않고 service 파일로 분리한다.
- 타입은 types 폴더에 둔다.
- mock data는 data 폴더에 둔다.

## 4. 작업 순서
- 먼저 현재 구조를 분석한다.
- 그다음 수정 계획을 짧게 제시한다.
- 사용자가 허락한 범위만 수정한다.
- 수정 후 npm run build를 실행한다.
- 수정한 파일 목록과 기존 기능 영향 여부를 한국어로 보고한다.

## 5. Supabase 작업 원칙
- 먼저 테이블 목록을 조회한다.
- 기존 테이블 구조를 확인한다.
- 바로 테이블 생성이나 수정은 하지 않는다.
- RLS 정책은 사용자가 요청할 때만 수정한다.
- 학년, 과목, 대단원, 중단원 데이터는 서비스 파일을 통해 조회한다.

## 6. 디자인 원칙
- 1%연산 같은 학생용 게임형 UI를 참고한다.
- 복사하지 말고 툰스쿨에 맞게 재해석한다.
- 배경은 연보라, 하늘색, 민트 계열의 어린이용 배경을 사용한다.
- 원색 남발을 피한다.
- 폰트는 둥글고 부드럽게 보이도록 한다.
- 태블릿 가로 화면을 우선한다.
- 왼쪽 선택, 오른쪽 결과/다음 단계 구조를 우선 검토한다.
