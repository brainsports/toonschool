# ToonSchool Claude Development Rules



## 1\. 절대 수정 금지 영역



* 기존 `/toon` 에디터는 사용자가 명시적으로 요청하지 않는 한 수정하지 않는다.
* 다음 파일은 사용자의 명시적 요청이 없는 한 수정하지 않는다.

  * `ToonEditor.tsx`
  * `A4CoverPreview.tsx`
  * `ComicPreview.tsx`
  * `SpeechBubblePanel.tsx`
  * `SpeechBubbleOverlay.tsx`
  * `ComicCutPanel.tsx`
* Supabase에서 다음 작업은 절대 실행하지 않는다.

  * 테이블 삭제
  * `DROP`
  * `TRUNCATE`
  * 대량 `UPDATE`
  * 대량 `DELETE`
* Gemini API 설정, Supabase 설정, 인증 설정은 사용자의 요청 없이는 수정하지 않는다.
* 운영 데이터와 운영 DB는 사용자가 명확하게 승인하지 않는 한 직접 변경하지 않는다.
* 요청과 관련 없는 기존 기능을 임의로 리팩터링하거나 구조 변경하지 않는다.
* 만화 생성 관련 코드, DB, RPC, Edge Function, 생성 횟수, 예약·확정·해제 로직은 보호 대상이다. 먼저 읽기 전용으로 원인을 분석하고 수정 대상·영향 범위·검증 방법을 사용자에게 보고한 뒤, 명시적 승인을 받아야만 수정할 수 있다. 승인 전에는 코드 변경, SQL 실행, 마이그레이션, Git commit·push, main 병합, 배포를 하지 않는다.



## 2\. 학생 UI 작업 원칙



* 학생 화면 작업은 가능한 한 `/student` 관련 파일 범위에서 처리한다.
* 학생 UI는 태블릿 가로 화면을 우선하여 설계한다.
* 초등학생용 게임형 학습 앱의 친근한 톤앤매너를 유지한다.
* 흰 배경 중심의 일반적인 업무용 웹폼 형태를 피한다.
* 한 화면에서 학생이 다음에 눌러야 할 버튼이 분명하게 보이도록 한다.
* 문구는 짧고 쉽고 크게 작성한다.
* 카드, 버튼, 배경, 보상, 진행률은 통일된 디자인을 유지한다.
* 배경은 연보라, 하늘색, 민트 계열을 우선 사용한다.
* 원색을 지나치게 많이 사용하지 않는다.
* 폰트와 버튼은 둥글고 부드럽게 보이도록 한다.
* 왼쪽 선택, 오른쪽 결과 또는 다음 단계 구조를 우선 검토한다.
* 기존 디자인 시스템과 캐릭터 스타일을 가능한 한 유지한다.



## 3\. 모듈화 원칙



* 한 파일은 가능한 한 600행을 넘지 않게 한다.
* 400행을 넘으면 컴포넌트 또는 서비스 분리 가능성을 검토한다.
* 페이지 파일에는 전체 흐름과 조합만 둔다.
* 실제 UI는 `components` 폴더로 분리한다.
* Supabase 조회 및 저장 코드는 페이지 안에 직접 길게 작성하지 않고 `service` 파일로 분리한다.
* 타입은 `types` 폴더 또는 관련 타입 파일에 둔다.
* mock data는 `data` 폴더 또는 별도 mock 파일에 둔다.
* 단순 수정 작업에서는 불필요한 파일 분리나 대규모 리팩터링을 하지 않는다.
* 사용자가 요청한 작업보다 리팩터링 범위가 커질 경우 먼저 중단하고 보고한다.



## 4\. 작업 유형 분류



모든 작업은 시작할 때 아래 기준으로 구분한다.



### 4.1 단일 페이지 독립 수정



다음과 같은 작업을 의미한다.



* 문구 변경
* 오탈자 수정
* 글자 크기 변경
* 색상 변경
* 여백 변경
* 정렬 변경
* 카드 크기 변경
* 버튼 위치 변경
* 이미지 교체
* 단순 CSS 수정
* 한 페이지 안에서만 끝나는 레이아웃 수정
* 다른 페이지, 데이터, 권한, 저장 결과에 영향을 주지 않는 수정



### 4.2 연동 작업



다음 중 하나라도 해당하면 연동 작업으로 판단한다.



* 다른 페이지와 결과가 연결되는 작업
* 학생 화면과 교사 화면이 연결되는 작업
* 학생·교사·기관·중간관리자·수퍼관리자 화면이 서로 연결되는 작업
* 목록 화면과 상세 화면이 연결되는 작업
* 저장·수정·삭제 결과가 다른 화면에 표시되는 작업
* Supabase 데이터 조회 또는 저장과 연결되는 작업
* 로그인 또는 권한과 연결되는 작업
* 점수, 레벨, 보상, 출석, 성장 현황과 연결되는 작업
* 작품 제출, 평가, 알림, 공유와 연결되는 작업
* 공통 컴포넌트 또는 공통 상태가 여러 페이지에 영향을 주는 작업
* 라우팅 또는 메뉴 구조가 여러 화면에 영향을 주는 작업
* 한 화면의 변경 결과가 다른 화면에서 반드시 확인되어야 하는 작업



### 4.3 고위험 작업



다음 작업은 고위험 작업으로 판단한다.



* Supabase 테이블, 컬럼, 정책, RPC, Trigger 변경
* DB 마이그레이션
* 인증 및 권한 구조 변경
* 결제 또는 이용권 로직 변경
* 점수·레벨·보상 계산식 변경
* 운영 데이터 변경
* Edge Function 변경
* 공통 모듈 변경으로 여러 핵심 화면에 영향을 주는 작업
* 기존 사용자 데이터 손실 가능성이 있는 작업



고위험 작업은 사용자의 명시적 지시가 없으면 실행하지 않는다.



## 5\. 기본 작업 원칙



* 사용자가 요청한 범위만 수정한다.
* 관련 없는 페이지와 기능은 조사하거나 수정하지 않는다.
* 작업 범위가 명확하면 장시간 전체 프로젝트를 분석하지 않는다.
* 간단한 UI 수정에 과도한 토큰과 시간을 사용하지 않는다.
* 기존 구현을 최대한 활용하고 최소 변경으로 해결한다.
* 작업 도중 불필요한 중간 보고로 멈추지 않는다.
* 사용자가 계획 보고를 요청한 경우에만 계획을 먼저 보고한다.
* 별도 요청이 없으면 구현부터 배포까지 한 번에 끝낸다.
* 성공한 작업 이후 더 좋은 구조를 찾기 위한 추가 탐색이나 리팩터링을 하지 않는다.
* 완료 후 한 번만 짧게 최종 보고한다.



## 6\. 단일 페이지 독립 수정 실행 절차



단일 페이지 독립 수정은 다음 순서로 진행한다.



1. 요청 화면과 직접 관련된 파일만 찾는다.
2. 요청한 부분만 최소한으로 수정한다.
3. 관련 없는 파일은 열거나 수정하지 않는다.
4. 문법 오류, 타입 오류, 빌드 오류만 확인한다.
5. `npm run build`를 1회 실행한다.
6. 작업 시작 전과 커밋 전에 `git status --short`를 확인한다.
7. 요청과 관련된 파일만 커밋한다.
8. 기능 브랜치에 Git 커밋한다.
9. 기능 브랜치를 원격에 push한다.
10. main 브랜치에 병합한다.
11. `origin/main`에 push한다.
12. Vercel Production 배포 완료 상태만 확인한다.
13. 별도의 Gemini 검증과 실제 브라우저 검증은 생략한다.
14. 결과 화면은 사용자가 직접 확인한다.
15. 작업을 즉시 종료하고 최종 보고한다.



단일 페이지 독립 수정에서 다음 작업은 하지 않는다.



* 전체 사이트 회귀 테스트
* 전체 메뉴 탐색
* 모든 사용자 역할 테스트
* 새 테스트 계정 생성
* 관련 없는 모바일·태블릿·데스크톱 전체 테스트
* 장문의 분석 보고서 작성
* 별도 검증용 스크립트 대량 생성
* 성공 이후 추가 리팩터링



## 7\. 연동 작업 실행 절차



연동 작업은 다음 순서로 진행한다.



1. 요청 기능과 직접 연결된 페이지, 데이터, 상태, 권한 범위를 확인한다.
2. 전체 프로젝트가 아니라 관련 연동 흐름만 조사한다.
3. 요청한 기능을 구현한다.
4. 관련된 시작 화면부터 결과 화면까지 검증한다.
5. 저장, 수정, 삭제, 조회 결과가 연결된 화면에 정상 반영되는지 확인한다.
6. 학생·교사·관리자 등 관련 역할 사이의 연동이 있다면 해당 역할만 확인한다.
7. 관련 흐름의 오류가 없으면 `npm run build`를 실행한다.
8. 작업 시작 전과 커밋 전에 `git status --short`를 확인한다.
9. 요청과 관련된 파일만 커밋한다.
10. 기능 브랜치에 Git 커밋한다.
11. 기능 브랜치를 원격에 push한다.
12. main 브랜치에 병합한다.
13. `origin/main`에 push한다.
14. Vercel Production 배포 완료를 확인한다.
15. 운영 환경에서 요청과 직접 관련된 연동 흐름만 다시 확인한다.
16. 전체 사이트 회귀 테스트는 하지 않는다.
17. 작업을 종료하고 최종 보고한다.



연동 작업의 검증 예시는 다음과 같다.



* 학생이 작품을 제출하면 교사 관리 목록에 나타나는지 확인
* 교사가 평가하면 학생 성장 현황과 점수에 반영되는지 확인
* 학생 점수 변경이 레벨과 보물지도에 반영되는지 확인
* 학급 배정 변경이 학생·교사 화면에 함께 반영되는지 확인
* 저장한 데이터가 목록과 상세 화면에 동일하게 표시되는지 확인
* 알림 발송 결과가 대상 사용자 알림함에 표시되는지 확인



연동 작업에서도 다음은 하지 않는다.



* 요청과 관련 없는 페이지 검증
* 모든 사용자 역할 전체 테스트
* 전체 메뉴 탐색
* 관련 없는 오류 조사
* 성공 이후 추가 탐색
* 장문의 검증 보고서 작성



## 8\. 검증 담당 원칙



* 단일 페이지 독립 수정은 별도의 Gemini 검증을 실행하지 않는다.
* 연동 작업은 Gemini 또는 실제 브라우저를 사용하여 관련 연동 흐름을 반드시 검증한다.
* Gemini는 구현 담당이 아니라 검증 담당으로 사용한다.
* Gemini는 요청받은 연동 범위만 확인한다.
* Gemini는 관련 없는 페이지와 기능을 검증하지 않는다.
* Gemini는 새로운 테스트 계정을 만들지 않는다.
* 제공된 테스트 계정이 있으면 해당 계정만 사용한다.
* 검증 성공 후 추가 탐색 없이 종료한다.
* 검증 결과는 성공 또는 실패와 핵심 근거만 짧게 보고한다.



## 9\. Git 및 배포 원칙



사용자가 별도로 제외하지 않는 한 모든 작업은 다음까지 완료한다.



1. Git 커밋
2. 기능 브랜치 push
3. main 브랜치 병합
4. `origin/main` push
5. Vercel Production 배포 확인



작업 시작 전과 커밋 전에는 반드시 다음을 확인한다.



```bash

git status --short



커밋에는 요청과 직접 관련된 파일만 포함한다.



다음 파일은 커밋하지 않는다.



* 임시 스크립트
* 테스트 캡처
* 브라우저 캡처
* 분석 메모
* 로그 파일
* 일회성 검증 파일
* 임시 JSON 파일
* 임시 SQL 파일
* 디버깅 파일
* 요청과 관련 없는 기존 변경 파일



작업 트리에 사용자의 기존 변경 파일이 있으면 임의로 삭제, 초기화, 복원하지 않는다.



사용자의 기존 변경과 충돌할 가능성이 있으면 해당 파일은 건드리지 않고 보고한다.



## 10\. Vercel 배포 원칙



* 사용자가 별도로 제외하지 않는 한 main 병합과 Vercel Production 배포까지 완료한다.
* Vercel Preview 배포만 하고 종료하지 않는다.
* Production 배포가 main 최신 커밋을 가리키는지 확인한다.
* 단일 페이지 독립 수정은 Production 배포 완료 상태까지만 확인한다.
* 연동 작업은 Production 배포 후 관련 연동 흐름을 운영 환경에서 확인한다.
* 배포가 실패하면 원인을 확인하고 요청 범위 안에서 수정한다.
* 배포 성공 후 관련 없는 로그를 장시간 분석하지 않는다.



## 11\. Supabase 작업 원칙



* Supabase 작업은 먼저 기존 테이블과 구조를 확인한다.
* 바로 테이블 생성이나 수정부터 하지 않는다.
* 기존 서비스 파일과 쿼리 구조를 먼저 확인한다.
* RLS 정책은 사용자가 요청할 때만 수정한다.
* DB 마이그레이션은 사용자의 명시적 승인 없이 운영에 적용하지 않는다.
* 학년, 과목, 대단원, 중단원 데이터는 서비스 파일을 통해 조회한다.
* 서비스 역할 키 또는 민감한 키를 코드에 직접 작성하지 않는다.
* 운영 DB를 테스트 목적으로 임의 변경하지 않는다.
* 테스트 데이터 생성이 필요하면 사용자에게 제공된 테스트 계정과 기존 데이터를 우선 사용한다.
* 새로운 테스트 계정은 사용자의 요청 없이 만들지 않는다.



## 12\. 빌드 및 테스트 원칙



* 모든 코드 수정 후 기본적으로 `npm run build`를 1회 실행한다.
* 같은 결과를 확인하기 위해 빌드를 반복 실행하지 않는다.
* 빌드가 성공하면 단순 수정에서 추가 테스트를 확대하지 않는다.
* 빌드 실패 시 요청과 직접 관련된 오류인지 먼저 확인한다.
* 기존의 관련 없는 오류라면 임의로 전체 프로젝트를 수정하지 않고 보고한다.
* 단위 테스트가 이미 있고 요청 기능과 직접 관련된다면 해당 테스트만 실행한다.
* 전체 테스트 스위트는 고위험 작업이나 사용자의 명시적 요청이 있을 때만 실행한다.



## 13\. 작업 시간 및 토큰 절약 원칙



* 단순 UI/CSS 수정은 가능한 한 10\~15분 안에 구현한다.
* 간단한 수정에서 전체 파일 구조를 장시간 탐색하지 않는다.
* 같은 파일을 반복해서 여러 번 읽지 않는다.
* 이미 확인한 내용을 다시 조사하지 않는다.
* 불필요한 보고서, 계획서, 검증 문서를 만들지 않는다.
* 작업 과정 전체 로그를 최종 보고에 포함하지 않는다.
* 명령어 실행 기록을 그대로 길게 복사하지 않는다.
* 요청 성공 여부를 판단하는 데 필요한 최소한의 작업만 수행한다.
* 사용자의 요청이 완료되면 즉시 종료한다.



## 14\. 금지 사항



* 단일 페이지 수정인데 전체 사이트를 검증하지 않는다.
* 연동 작업인데 관련 연동 검증을 생략하지 않는다.
* 관련 없는 기능을 임의로 개선하지 않는다.
* 사용자가 요청하지 않은 기능을 추가하지 않는다.
* 요청 없이 운영 DB를 수정하지 않는다.
* 요청 없이 새로운 계정을 만들지 않는다.
* 요청 없이 기존 파일을 삭제하지 않는다.
* 사용자의 기존 Git 변경을 임의로 되돌리지 않는다.
* 임시 파일을 Git에 포함하지 않는다.
* 작업 성공 후 추가 리팩터링을 하지 않는다.
* 작업 도중 중간 보고만 하고 종료하지 않는다.
* main 병합과 Vercel 배포를 남겨 둔 채 완료라고 보고하지 않는다.

## 16\. 최종 보고 형식



최종 보고는 한국어로 짧게 작성한다.



다음 항목만 포함한다.



* 수정한 내용
* 수정한 파일
* 커밋 SHA
* 기능 브랜치 push 여부
* main 병합 여부
* Vercel Production 배포 여부
* 검증 여부
* 남은 문제



단일 페이지 독립 수정의 검증 항목은 다음처럼 작성한다.



```text

검증: 별도 브라우저 검증 생략 — 사용자가 결과 화면 직접 확인

```



연동 작업의 검증 항목은 다음처럼 작성한다.



```text

검증: 관련 연동 흐름 확인 완료

```



다음 내용은 최종 보고에 포함하지 않는다.



* 전체 명령어 기록
* 긴 분석 과정
* 관련 없는 조사 내용
* 반복적인 설명
* 불필요한 표
* 장문의 회고
